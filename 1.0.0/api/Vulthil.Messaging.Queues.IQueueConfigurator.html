SubscribeAll<TInterface>(Assembly, string?)

Discovers every concrete (non-abstract, non-interface) type in assembly that is assignable to TInterface, and calls Subscribe<TMessage>(string?) for each. Pair with a polymorphic AddConsumer<TConsumer>() (where TConsumer : IConsumer<TInterface>) to dispatch all implementers through one consumer.

Subscribe<TMessage>(string?)

Subscribes this queue to receive deliveries of the concrete message type TMessage. At topology setup time, the queue is bound to TMessage's exchange with the supplied routingKey pattern (the broker uses this to filter; the worker does not re-filter). Abstract types and interfaces are rejected — those have no exchange; use SubscribeAll<TInterface>(Assembly, string?) or call Subscribe<TMessage>(string?) for each concrete implementer instead.

IQueueConfigurator Subscribe<TMessage>(string? routingKey = null) where TMessage : class

Parameters

routingKey string

The binding pattern. When null, the broker receives an empty string — fanout/headers exchanges ignore it; direct exchanges only deliver messages with an empty published routing key; topic exchanges match no patterns. For non-empty needs, supply a specific pattern (e.g. "order.created" for direct, "order.*" for topic).

UseSingleActiveConsumer()

Declares the queue with RabbitMQ's single active consumer feature: only one consumer is active at a time while additional consumers stand by and take over on failure. This preserves per-queue order across load-balanced consumer instances — extending the in-process partitioner's ordering guarantee to multiple instances — at the cost of throughput scale-out for the queue. Partitioned queues enable this automatically.
