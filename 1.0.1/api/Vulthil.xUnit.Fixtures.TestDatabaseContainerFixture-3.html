Fixture that wraps a database container, applies EF Core migrations, and supports Respawn-based data resets between tests. When owned by a ContainerHost, CreateScope(string) gives every consuming factory its own uniquely named database on the shared server, so parallel test classes never see each other's data.

ConfigureServices(IServiceCollection)

Registers or replaces services for this container in the test host. Invoked through ConfigureTestServices, so it runs after the application's own registrations and can decorate or replace them (for example, repointing a DbContext at the containerized service).

public virtual void ConfigureServices(IServiceCollection services)

Parameters

services IServiceCollection

Remarks

The base implementation decorates the application's DbContextOptions<TContext> registration to ignore ManyServiceProvidersCreatedWarning. With shared containers every test class fixture (and every per-test WithWebHostBuilder clone) builds its own host, and therefore its own EF Core internal service provider, so large suites trip that warning even though each host is correct. Overrides should call base to preserve this behavior.

CreateScope(string)

Creates a scope view backed by its own database on this server, named after scopeId. The view creates the database when it initializes, applies migrations through MigrateDatabase(IServiceProvider), resets it with Respawn between tests, and drops it (best-effort) when disposed. The server container itself keeps running for other scopes.

DropDatabaseAsync(string)

Drops the database databaseName on the server. Invoked best-effort when a scope is disposed; failures are reported as diagnostics rather than test failures, because the server container is torn down with the host anyway. The default issues engine-appropriate DDL based on DbAdapter — PostgreSQL terminates lingering (pooled) sessions with WITH (FORCE), SQL Server switches the database to single-user first — so consumers normally never override this.

ResetAsync(IServiceProvider)

Resets the resource to a clean state after each test, with the application's root service provider so a resource that needs the app's services — for example a Cosmos store recreating its database through the registered DbContext — can resolve them from a fresh scope. Resources that need no services ignore the argument. One-time setup that must run before the first test instead belongs in IStartupResource.
