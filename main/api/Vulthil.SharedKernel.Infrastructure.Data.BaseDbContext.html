ExecuteInTransactionAsync<TResult>(Func<CancellationToken, Task<TResult>>, Func<TResult, bool>, CancellationToken)

Runs operation inside a database transaction, committing only when shouldCommit returns true for the produced result and rolling back otherwise (or when operation throws), using the store's execution strategy.

public Task<TResult> ExecuteInTransactionAsync<TResult>(Func<CancellationToken, Task<TResult>> operation, Func<TResult, bool> shouldCommit, CancellationToken cancellationToken = default)

Type Parameters

TResult

Remarks

Use this overload to roll back on a returned failure — for example a failed Result from a command handler — rather than only on a thrown exception. The same execution-strategy and ambient-transaction semantics as ExecuteInTransactionAsync<TResult>(Func<CancellationToken, Task<TResult>>, CancellationToken) apply; when a transaction is already active the outer scope owns the commit and shouldCommit is not consulted.

ExecuteInTransactionAsync<TResult>(Func<CancellationToken, Task<TResult>>, CancellationToken)

Runs operation inside a database transaction, committing on success and rolling back on failure, using the store's execution strategy so it is compatible with a retrying execution strategy.

public Task<TResult> ExecuteInTransactionAsync<TResult>(Func<CancellationToken, Task<TResult>> operation, CancellationToken cancellationToken)

Parameters

operation Func<CancellationToken, Task<TResult>>

The work to run inside the transaction.

cancellationToken CancellationToken

A token to observe for cancellation.

Returns

Task<TResult>

The result produced by operation.

Type Parameters

TResult

The type produced by operation.

Remarks

Prefer this over manually pairing BeginTransactionAsync(CancellationToken) with a commit when a retrying execution strategy may be configured (e.g. the EF Core default): a bare user-initiated transaction is rejected under a retrying strategy, whereas this wraps the whole begin/operation/commit as one retriable unit. A transient-fault retry re-runs operation from a clean change-tracker state, so it must be idempotent. If a transaction is already active, operation simply joins it (the outer scope owns the commit), so this composes with an outer caller that already opened a transaction.
