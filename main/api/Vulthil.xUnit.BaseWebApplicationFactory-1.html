Remarks

Migrations run from an IHostedService registered at the front of the host's hosted-service list, so the schema exists before the application's own background services start. The migration step only applies migrations that are still pending and tolerates a concurrent migrator, so an application that migrates itself on startup keeps ownership and the factory never interferes with the application's own migration logic. Use a derived factory as an Xunit.IClassFixture<TFixture> (or collection fixture) so its containers are started once and shared across the tests in that scope, while BaseIntegrationTestCase<TFactory, TEntryPoint> resets database state between tests. To share containers across many parallel test classes, register them on a ContainerHost assembly fixture instead and pass it to the BaseWebApplicationFactory(ContainerHost) constructor; the factory then consumes every host container through a per-factory scope view (an isolated database, virtual host, ...) so test classes run in parallel against shared containers without interfering.
